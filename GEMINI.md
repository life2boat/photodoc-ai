GEMINI.md — Конвенция проекта PhotoDoc AI
1. Назначение и Стек
Этот документ определяет границы ответственности и правила работы ИИ-агента с проектом PhotoDoc AI.
Технологический стек:

Frontend: React + Vite + Tailwind CSS + Shadcn UI.

Backend: Python + FastAPI + SQLAlchemy + AsyncOpenAI (легковесный бэкенд).

AI-модели: Использование внешних облачных API (VSEGPT) вместо локальных тяжелых моделей.

2. Роль Агента и Границы (Guardrails)
Агент является техническим исполнителем, а не архитектором.

Разрешено: Изменять тексты, стили (Tailwind), создавать новые UI-компоненты по образцу, писать документацию, исправлять мелкие баги логики.

ЗАПРЕЩЕНО без явного подтверждения:

Изменять конфигурационные файлы: vite.config.js, tailwind.config.js, package.json.

Менять настройки CORS в main.py.

Переписывать пути к весам нейросетей (.pth файлы).

Менять структуру папок frontend/src и backend/app.

3. Правила работы с кодом (Code Standards)
Frontend (React/Vite)
Пути: Использовать относительные пути для импортов (например, ../../lib/utils), если не подтверждена корректная работа алиаса @.

Компоненты: Новые UI-компоненты добавлять строго в src/components/ui.

Логика: При использовании useEffect для работы с файлами всегда предусматривать очистку памяти через URL.revokeObjectURL.

Backend (FastAPI/Python)
Отступы: Строго 4 пробела. Ошибки IndentationError недопустимы.

Асинхронность: Все эндпоинты обработки изображений должны быть async def.

Управление памятью: Использовать BackgroundTasks для удаления временных файлов после отправки ответа.

4. Инструкции для MCP-инструментов
Если доступны MCP-серверы, использовать их следующим образом:

Filesystem MCP: Ограничить доступ только директорией проекта. Не сканировать системные папки.

Playwright MCP: Использовать для автоматического тестирования связки фронтенд-бэкенд.

Сценарий теста: Загрузка фото -> Нажатие кнопки -> Проверка появления результата.

Fetch MCP: Использовать только для поиска актуальной документации по библиотекам (например, lucide-react или fastapi).

5. Протокол внесения изменений
Перед выполнением любой задачи агент обязан:

Прочитать текущее состояние затронутых файлов.

Сформулировать план изменений в 2-3 пунктах.

Получить подтверждение пользователя.

После внесения изменений убедиться, что серверы Vite и Uvicorn запускаются без ошибок.

6. Безопасность
Никогда не записывать API-ключи или пароли в код. Использовать только .env файлы.

Не изменять системные переменные окружения пользователя.

# Описание проекта: PhotoDoc AI

## Роль ИИ-агента
Ты — Senior Fullstack Developer. Твоя задача — помогать развивать проект PhotoDoc AI, соблюдая стандарты и используя MCP-инструменты. Уровень английского владельца проекта — B2, поэтому техническую документацию и комментарии пиши понятным, но профессиональным языком.

## Архитектура
- **Frontend:** React + Vite, Tailwind CSS. Лежит в папке `/frontend`.
- **Backend:** Python + FastAPI. Лежит в папке `/backend`.
- **База данных:** Supabase.

## Правила работы (Guardrails)
1. **Context7:** Перед внедрением новых компонентов всегда сверяйся с актуальной документацией через Context7 MCP.
2. **Filesystem:** Анализируй файлы проекта через Filesystem MCP.
3. **Безопасность:** Никогда не сохраняй секретные ключи в коде.
## 8. Имитация MCP-процессов (Ручной режим)
Поскольку нативная интеграция MCP ограничена, мы используем следующий протокол:

- **GitHub Workflow:** Агент НИКОГДА не делает `git push`. Агент только генерирует осмысленные сообщения для коммитов (Conventional Commits) на основе предоставленного `diff` или измененных файлов. Коммит и пуш делает человек.
- **Chrome DevTools (UI/UX Debug):** Агент выступает как CSS-консультант. Человек предоставляет HTML/CSS проблемного блока и скриншот/описание ошибки. Агент предлагает исправления Tailwind-классов согласно `DEVTOOLS-CHECKLIST.md`.
- **Playwright (E2E Тесты):** Агент генерирует тесты `.spec.ts` для проверки связки фронтенд-бэкенд (загрузка фото -> отправка -> success state) и помогает чинить их на основе логов ошибок из терминала.
